# План разработки программного обеспечения для автоматизации работы предприятия общественного питания

## 2. Разработка клиентской части веб-приложения

### 2.1 Разработка макетов страниц
Процесс проектирования интерфейса ориентирован на создание интуитивно понятной среды для различных категорий пользователей (сотрудников и клиентов). Разработаны следующие макеты:
*   **Модуль аутентификации (LoginPage)**: Реализация формы входа с валидацией данных на стороне клиента. Обеспечивает безопасный доступ к системе через проверку JWT-токенов.
*   **Интерактивная карта зала (FloorMap)**: Визуальное представление расположения столов с динамическим изменением их состояния. Реализована логика drag-and-drop для администратора и визуализация занятости для персонала.
*   **Интерфейс обслуживания (WaiterInterface)**: Многофункциональная панель для официантов, включающая выбор блюд из цифрового меню, привязку заказа к конкретному столу и отслеживание статуса готовности.
*   **Кухонный терминал (KitchenInterface)**: Оптимизированный интерфейс для поваров, отображающий очередь заказов в реальном времени с приоритезацией по времени поступления.
*   **Панель администратора (ManagementInterface & UserManagement)**: Модули для управления кадровым составом, редактирования позиций меню и настройки параметров заведения.
*   **Аналитический модуль (Statistics)**: Дашборд с графиками и диаграммами, отображающий ключевые показатели эффективности (выручка, популярные блюда, загрузка зала).
*   **Клиентский портал (ClientPortal)**: Внешний интерфейс для бронирования столов и предварительного ознакомления с ассортиментом.

### 2.2 Описание пользовательского меню
Система управления доступом базируется на ролевой модели (Role-Based Access Control — RBAC):
*   **Администратор**: Полный доступ к системным настройкам, финансовой отчетности и управлению учетными записями.
*   **Менеджер**: Управление сменами персонала, редактирование меню, разрешение конфликтных ситуаций с бронированием.
*   **Официант**: Работа с активными заказами, изменение статусов столов ("Свободен", "Занят", "Забронирован", "Грязный"), печать чеков.
*   **Повар**: Управление процессом приготовления (прием заказа в работу, отметка о готовности).
*   **Клиент**: Просмотр доступного времени для бронирования и отправка заявок.

### 2.3 Верстка страниц
Технологический стек верстки включает передовые инструменты разработки:
*   **React 19**: Использование компонентного подхода для обеспечения переиспользуемости кода и высокой скорости рендеринга через Virtual DOM.
*   **Material UI (MUI 6)**: Применение библиотеки готовых компонентов, соответствующих концепции Material Design, что гарантирует визуальную целостность интерфейса.
*   **TypeScript**: Обеспечение строгой типизации данных, что минимизирует количество ошибок на этапе разработки и упрощает поддержку кода.
*   **State Management**: Использование React Context API для управления глобальным состоянием (данные авторизации, текущая роль пользователя).

## 3. Разработка серверной части веб-приложения

### 3.1 Описание атрибутов таблиц и их типов
База данных спроектирована в СУБД PostgreSQL. Основные сущности включают:
*   `users`: `id (SERIAL)`, `username (VARCHAR)`, `password_hash (TEXT)`, `role_id (INT)`, `first_name`, `last_name`.
*   `tables`: `id (SERIAL)`, `number (INT)`, `seats (INT)`, `status (VARCHAR)`, `x, y (INT)` для позиционирования.
*   `menu_items`: `id (SERIAL)`, `name (VARCHAR)`, `price (INT)`, `category (VARCHAR)`, `image_url (TEXT)`.
*   `bookings`: `id (SERIAL)`, `table_id (INT)`, `guest_name (VARCHAR)`, `phone (VARCHAR)`, `booking_date (DATE)`.
*   `orders`: `id (SERIAL)`, `table_id (INT)`, `status (VARCHAR)`, `total (INT)`, `created_at (TIMESTAMP)`.

### 3.2 Описание связи данных, построение схемы данных
Архитектура БД соответствует требованиям третьей нормальной формы (3NF):
*   **Связи 1:M (Один ко многим)**: Одна роль — много пользователей; один стол — много бронирований; один заказ — много позиций.
*   **Связи M:M (Многие ко многим)**: Реализованы через связующие таблицы, например, `booking_time_slots` для связи бронирования с несколькими временными интервалами.
*   **Целостность**: Использование ограничений `FOREIGN KEY`, `NOT NULL` и `UNIQUE` для предотвращения дублирования данных и обеспечения логической связанности.

### 3.3 Описание программных модулей
Серверная часть реализована на платформе Node.js с использованием фреймворка Hono:
*   **Auth Module**: Реализация стратегии аутентификации на основе JSON Web Tokens (JWT). Хеширование паролей с использованием алгоритма bcrypt.
*   **Resource Controllers**: Набор модулей (controllers) для обработки CRUD-запросов к столам, меню, заказам и пользователям.
*   **Middleware Layer**: Промежуточное ПО для проверки прав доступа, логирования запросов и обработки CORS.
*   **File Storage Service**: Модуль для обработки загрузки изображений блюд и их отдачи клиенту.

### 3.4 Тестирование веб-приложения
Программа тестирования включает несколько уровней:
*   **Unit-тестирование**: Проверка отдельных функций (расчет стоимости заказа, валидация формата телефона).
*   **Integration Testing**: Проверка взаимодействия между API-эндпоинтами и базой данных.
*   **End-to-End (E2E) Testing**: Сценарное тестирование основных путей пользователя (от входа в систему до закрытия заказа).
*   **Manual Testing**: Проверка соответствия интерфейса макетам и тестирование адаптивности на реальных устройствах.

### 3.5 Размещение веб-приложения на хостинге
Стратегия деплоя и эксплуатации:
*   **Docker-контейнеризация**: Создание образов для frontend-части (на базе Nginx), backend-части и базы данных PostgreSQL.
*   **Orchestration**: Использование Docker Compose для быстрого развертывания всей инфраструктуры одной командой.
*   **Environment Configuration**: Разграничение настроек для сред разработки (development) и эксплуатации (production) через переменные окружения.
*   **Scalability**: Готовность архитектуры к горизонтальному масштабированию при увеличении нагрузки.
